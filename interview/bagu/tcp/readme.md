# TCP/IP

- _FP(First Paint)_ 首次渲染所花的时间
  从页面加载到首次开始绘制的时长
  - _TTFB_(Time To First Byte) 从浏览器发起请求（用户点击链接或刷新页面）开始，到浏览器接收到服务器返回的第一个字节数据所经历的时间
    - 1. DNS解析时间（ip查找+负载均衡）
    - 2. TCP（三次握手）/TLS（加密）
    - 3. server执行时间（慢查询）
  - _响应下载时间_
    - 从收到第一个字节开始，到整个 HTML 文件完全下载完毕的时间
  - _解析与构建层_
    - HTML DOM树构建 + CSSOM树构建 + 渲染树构建
  - _布局与绘制层_
    - 布局树构建 + 首次渲染时间

  FP = TTFB + 响应下载时间 + HTML DOM树构建时间+CSSOM树构建时间+渲染树构建时间+布局树+首次渲染时间

  前端的性能标准 页面加载的速度,页面的打开速度是关键指标
  影响 用户留存率 付费欲望 PV(Page View 页面浏览量) UV(Unique Visitor 独立访客数)

- _网络加载速度_
  网络优化，OSIC七层架构 IP TCP/UDP HTTP TLS
  - HTTP是基于TCP/IP协议的应用层协议
  - 如何去优化web性能，轻松定位web问题

- _一个数据包的旅程_
  互联网，实际是一套理念和协议（不同版本）组成的体系架构
  互联网中的数据是通过数据包来传输的
  如果数据比较大？ 利用带宽、多路复用（多个数据包同时传输）
  数据拆分成很多小数据包来传输，提升带宽的利用率和并发效率
- 二进制数据帧
  互联网中的数据是通过二进制数据帧来传输的
  每个数据帧都有固定的大小（通常是 1500 字节）
  每个数据帧都包含了源 IP 地址、目的 IP 地址、协议类型、数据等信息

- ip地址（网络层） 将数据包送达目的主机
  Internet Protocol（IP） 计算机的地址
  丢失数据包，出错

- UDP（传输层）
  负责数据的到达
  UDP 机器上的某个应用（端口号）
  UDP 简单 快速 适合音视频的传输
  但是解决不了结构严格的HTML等web资源
  流程：IP 把包送到主机 -> 操作系统查看 UDP 头部中的目标端口 -> 把数据交给监听该端口的应用程序。
  比喻：大厦前台（操作系统）收到包裹，看标签上写着“8080 房间”，于是直接把包裹扔进 8080 房间的信箱。至于房间里的人收没收到，前台不管（UDP 的特性）。

- TCP
  浏览器请求、邮件要求数据传输可靠性的应用，如果使用UDP
  - 数据包在传输过程中容易丢失
    过期时间 _重传_
  - 数据包太多了，在不同的时间到达接收端
    给个序号 _组装_
  - 慢一点，完整到达
- 三次握手
  - 建立连接
    客户端向服务器发送一个 SYN(Synchronize Sequence Numbers（同步序列号）) 包，请求建立连接
  - A 请求 SEQ 设置一个同步值j
  - B 响应 ACK j+1 ACK = Acknowledgment (确认/收到)
  - A 确认收到 B 的响应

  发送数据以及接收数据的能力
  每一方的发送和接收都要有两次，握手四次
  第二次的时候可以响应对方的发送能力，第三次希望对方可以确认我们的发送能力

- 四次挥手
  A->B:可以关
  B->A:可以关
  B->A:你接收到了吗
  A->B:我收到了，关吧
