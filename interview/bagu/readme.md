- _浏览器的渲染进程运行在安全沙箱中_
  - 这个沙箱不是操作系统直接赠送的，而是浏览器利用操作系统提供的底层机制（如 Windows 的 Token 限制、Linux 的 Seccomp-BPF、macOS 的 Seatbelt）主动构建的。
  - 目的：
    将渲染进程的权限降到最低（最小权限原则）。
    它不能直接读写磁盘文件。
    它不能直接访问网络（必须通过 IPC 请求网络进程代劳）。
    它不能随意调用敏感的系统 API。
  - 效果：
    即使渲染进程加载了恶意网页代码被攻破，黑客也只能控制这个‘牢房’里的内容，无法窃取用户硬盘数据或控制系统，从而保证了浏览器的整体安全性。”
- _如果把整个页面加载比作装修房子：_
  浏览器主进程 = 项目经理（负责指挥、调度、验收）。
  渲染进程 = 施工队（负责画图、砌墙、刷漆，但在封闭工地里，不能随便出门买材料）。
  网络进程 = 采购员 + 物流司机。
  - 在“建立数据管道”之后：
    采购员（网络进程） 已经谈好了价格（握手），拿到了发货单（响应头）。
    现在，他开着卡车（数据管道），源源不断地把砖头水泥（HTML 字节流）从仓库（服务器）运到工地门口。
    施工队（渲染进程） 在门口接着砖头，立马就开始砌墙（解析 HTML），不用等整车货卸完。
    关键点：施工队（渲染进程）因为被关在沙箱里，严禁自己开车出去拉货，必须完全依赖采购员（网络进程）送货上门。

  - *OSI 七层协议*

      *TCP* (Transmission Control Protocol) 传输控制协议

      - 数据包 有大小上限的
      - 一个文件会分成好多个数据包 分批次 分通道传输
      - 数据包会丢失  重发 *TCP/IP* 协议会在数据链路层和网络层进行重发
        在 TCP 中，丢包会导致延迟增加和吞吐量下降，但数据不会丢失，文件不会损坏，网页内容依然完整

- _两种代理模式的区别_
  ![image copy 3.png](image%2Fimage%20copy%203.png)
  - 正向代理 = 代购
    你（客户端）想买国外的货（访问外网），但自己去不了。
    你找一个代购（代理服务器），告诉他你要买什么。
    代购去买回来给你。
    结果：国外商家只知道是“代购”买的，不知道是你买的。
  - 反向代理 = 公司前台
    客户（用户）来找某部门办事（访问服务）。
    客户只能见到前台（反向代理），把需求告诉前台。
    前台 internally 转交给具体的员工（后端服务器）处理，再把结果给客户。
    结果：客户只知道有这家公司，不知道具体是哪个员工（哪台服务器）处理的，甚至不知道员工坐在哪个工位（内网 IP）。

- *为什么我第一次在host文件中添加127.0.0.1 www.baidu.com是打不开baidu.com的，注释之后打的开，取消注释之后还是打的开*
- "DNS 缓存残留” 或 “浏览器预读取/连接池” 问题
- 浏览器的“独立”DNS 缓存 (最常见原因)
  现代浏览器（Chrome, Edge, Firefox）为了加速上网，自己有独立的 DNS 缓存，不完全依赖操作系统的 DNS 缓存。
  现象：即使你清除了系统的 DNS 缓存，浏览器内存里还存着：“哦，www.baidu.com 的真实 IP 是 xxx.xxx.xxx.xxx，我直接用这个 IP 连，不查 hosts 文件了。”
  为什么注释后能打开？ 当你注释掉规则时，如果浏览器缓存还没过期，它直接用了缓存里的真 IP，所以秒开。
  为什么取消注释后还能打开？ 同理，浏览器可能还在用之前的缓存，或者它建立了一个长连接（Keep-Alive），TCP 连接还没断开，你刷新页面时，它直接复用了这个已经连接到百度服务器的通道，根本没过问 hosts 文件。
